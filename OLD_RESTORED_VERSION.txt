## PROJECT OVERVIEW
This project is a Network Security Intelligence Tool that detects phishing threats using both database lookups and AI-powered heuristics.

================================================================================
## PART 1: WHAT THE PROJECT WAS ORIGINALLY
================================================================================

### Core Purpose:
The project started as a pure Machine Learning pipeline designed to train a model 
that could classify network traffic/URLs as "Phishing" or "Legitimate".

### The "Before" Architecture (Step-by-Step Process):

The original system was a linear pipeline orchestrated by `TrainingPipeline` 
(in `networksecurity/pipeline/training_pipeline.py`).

#### Step 1: Data Ingestion
- **Component:** `DataIngestion` (`networksecurity/component/data_ingestion.py`)
- **Action:** 
  1. Connected to MongoDB (`NetworkData` database).
  2. Exported collection data as a DataFrame.
  3. Saved raw data to a feature store.
  4. Split data into **Train** and **Test** sets.
- **Key Output:** `train.csv`, `test.csv`

#### Step 2: Data Validation
- **Component:** `DataValidation` (`networksecurity/component/data_validation.py`)
- **Action:**
  1. Checked if the data had the correct number of columns (based on schema).
  2. **Drift Detection:** Used `ks_2samp` (Kolmogorov-Smirnov test) to check if the 
     statistical properties of the test data were significantly different from the 
     training data.
  3. Generated a drift report.
- **Key Output:** Validated data files, Drift report.

#### Step 3: Data Transformation
- **Component:** `DataTransformation` (`networksecurity/component/data_transformation.py`)
- **Action:**
  1. Handled missing values using `KNNImputer` (K-Nearest Neighbors).
  2. Replaced target values (-1 for phishing) with standard labels (0/1).
  3. Created a preprocessing pipeline object.
- **Key Output:** Transformed Numpy arrays, `preprocessor.pkl` object.

#### Step 4: Model Training
- **Component:** `ModelTrainer` (`networksecurity/component/model_trainer.py`)
- **Action:**
  1. Trained multiple models:
     - Random Forest
     - Decision Tree
     - Gradient Boosting
     - Logistic Regression
     - AdaBoost
  2. **Hyperparameter Tuning:** Optimized parameters for each model.
  3. **Evaluation:** Compared models based on **F1 Score**.
  4. **Selection:** Picked the single best performing model.
  5. **Tracking:** Logged experiments (metrics, params, artifacts) to **MLflow** (hosted on DagsHub).
- **Key Output:** `model.pkl` (The final trained model).

#### Step 5: Artifact Syncing
- **Action:** Synced all generated artifacts (models, reports, data) to an **AWS S3** bucket for storage and versioning.

### Key Technologies Used:
- **MongoDB:** Primary data storage.
- **Scikit-Learn:** For model training and data preprocessing.
- **MLflow & DagsHub:** For tracking model performance and experiments.
- **AWS S3:** For artifact storage.
- **FastAPI:** Exposed simple endpoints (`/train`, `/predict`).

### Original Limitations:
- **Batch Only:** Designed for processing files, not real-time user queries.
- **No UI:** Interaction was purely through code or API calls.
- **Static:** Could only detect threats it had learned from the training data (no heuristic/pattern matching for new threats).

### STEP 3: Built Search Utility Functions
**File Created:** networksecurity/utils/search_utils.py

**What It Contains:**

1. **identify_input_type(input_str)**
   - Detects if user typed an IP, Domain, or URL
   - Example: "192.168.1.1" в†’ returns "ip"
   - Example: "google.com" в†’ returns "domain"

2. **calculate_risk_score(results)**
   - Takes database search results
   - Assigns points:
     * Found in phishing_links/combined_urls: +50 points
     * Found in domains: +30 points
     * Found in ips: +20 points
   - Returns score 0-100

3. **calculate_heuristic_score(input_str, input_type)**
   - Analyzes patterns even if NOT in database
   - Checks for:
     * Suspicious keywords (login, verify, secure): +10 per keyword
     * High-risk TLDs (.xyz, .top, .ru): +20
     * Unusually long URLs: +15
     * IP addresses in URLs: +25
     * Typosquatting (paypa1 vs paypal): +40
     * Dot-split evasion (ama.zon.com): +50
   - Returns {score: X, reasons: [...]}

**Why:**
These functions are the "brain" of the threat detection. The heuristic score is what 
makes your tool able to detect NEW phishing sites that aren't in your database yet.

## PART 8: SECURITY LOGIC REFINEMENTS (LATEST UPDATES)
================================================================================

### Issue 1: False Positives on Safe IPs
**Problem:** 
The system was flagging "Safe" IPs (labeled 'good' or '0' in the dataset) as risky simply because they existed in the database.
**Resolution:** 
Updated `calculate_risk_score` in `search_utils.py` to check the `result` or `label` field. Now, only entries explicitly marked as malicious contribute to the risk score.

### Issue 2: Low Risk Scores for Known Threats
**Problem:** 
Confirmed malicious IPs were receiving a low risk score (20/100), which appeared as "Safe" (Green) in the UI.
**Resolution:** 
Increased the risk weights for confirmed database hits. If an IP, Domain, or URL is found in the malicious database, it now receives a score of **100/100** (High Risk).

### Issue 3: Missed Brand Impersonation
**Problem:** 
Domains like `instagram-security-alert.org` were not being flagged because they contained the full brand name "instagram", which the system assumed was safe (to avoid flagging the official site).
**Resolution:** 
Implemented "Smart Brand Detection". The system now knows the official domains for major brands (e.g., `instagram.com`). If a URL contains a brand name but is **NOT** the official domain, it is flagged as **High Risk (+60 points)**.

### Issue 4: Undetected URL Chaining / Open Redirects
**Problem:** 
Attacks like `http://legitimate.com//http://phishing.com` were not being detected.
**Resolution:** 
Added heuristic checks for:
1. **Multiple Schemes**: Counts occurrences of `://`. If > 1, flags as "Multiple URL schemes detected" (+40 points).
2. **Obfuscation**: Checks for the `@` symbol in URLs (+30 points).
3. **New Keywords**: Added `alert`, `support`, `service`, `auth`, `security` to the suspicious keyword list.

These refinements have significantly improved the accuracy and reliability of the threat detection engine.
